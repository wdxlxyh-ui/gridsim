GridSim AI 自动化测试改进方案

让 IEC104 模拟器从"人类可用"升级到"AI 原生友好"——6 项基础设施改造，使 AI Agent 能自发现接口、自恢复错误、自录制回归

分支: feature/egc-scenario-testing 版本: v3.0.0 日期: 2026-06-14

📊 总览：6 项改进评估矩阵

# 改进项 当前状态 优先级 可行性 工作量 改动范围
1 API 自描述 (OpenAPI) ❌ 无 schema P0 高 中 (3-4天) 新增 1 文件 + 改路由注册
2 结构化错误契约 ❌ 裸字符串 {"error":"..."} P0 高 小 (1-2天) 重构 4 处 writeError

API 自描述能力 — OpenAPI Schema

暴露 /openapi.json，AI 一个请求获取全部接口定义
P0 · 最高优先级 可行性：高 工作量：3-4 天
现状分析
  • 路由用裸 mux.HandleFunc("/api/v1/instances", ...) 注册，无任何元数据
  • 无 swagger/openapi/json-schema 任何形式的接口描述文件
  • MCP Server 内部有工具 schema（mcp-go 的 mcp.WithString() 等），但仅限 stdio 通道，HTTP 层不可见
  • AI Agent 必须依赖外部文档（API.md）才能工作——文档与代码容易不同步
改造方案

用 Go struct tag 声明式描述路由，启动时自动生成 OpenAPI 3.0 JSON。

🔴 现状 — AI 需要读文档

AI Agent 工作流：
1. 读 API.md (600+ 行)
2. 人工理解每个端点
3. 手动构造请求
4. 无法验证参数合法性
→ 文档过期 = AI 出错

🟢 改造后 — AI 读 schema

AI Agent 工作流：
1. GET /openapi.json
2. 自动解析所有端点+参数+返回值
3. 按 schema 构造合法请求
4. 响应结构有保证
→ 代码即文档，永不过期
技术实现

func RegisterSpec(mux *http.ServeMux) {

结构化错误契约

从 {"error":"string"} 升级到带错误码、提示、候选值的结构化错误
P0 · 最高优先级 可行性：高 工作量：1-2 天
现状分析

当前 writeError 在 4 个文件中重复定义，返回裸字符串：

// cmd/gridsim/main.go:905  ← 4 处副本之一
func writeError(w http.ResponseWriter, status int, msg string) {
    writeJSON(w, status, map[string]string{"error": msg})
}

// 调用处示例 — AI 拿到错误后完全不知道该怎么修
writeError(w, 404, "scenario not found")
writeError(w, 400, "invalid JSON: unexpected end of JSON input")
writeError(w, 409, "port already in use")
改造方案

🔴 现在的错误

{
  "error": "scenario not found"
}

AI 收到后：
- 不知道有哪些可用场景
- 不知道错误码
- 无法自动纠错
- 只能问用户或放弃

🟢 改造后的错误

{
  "error": {
    "code": "SCENARIO_NOT_FOUND",
    "message": "场景 'peak_shift' 不存在",
    "hint": "使用 GET /api/v1/scenarios 查看可用场景",

- 错误码可程序化分支
- hint 告诉下一步操作

// pkg/errors/errors.go — 新增统一错误类型
type APIError struct {
    Code        string   `json:"code"`           // MACHINE_READABLE_CODE
    Message     string   `json:"message"`         // 人类可读描述
    Hint        string   `json:"hint,omitempty"`  // 修复建议
    Candidates  []string `json:"candidates,omitempty"`
    InstanceID  string   `json:"instance_id,omitempty"`
    Field       string   `json:"field,omitempty"` // 出错字段名
}

// 预定义错误码（部分）
const (
    ErrInstanceNotFound = "INSTANCE_NOT_FOUND"
    ErrPortInUse        = "PORT_IN_USE"
    ErrIOANotFound      = "IOA_NOT_FOUND"
    ErrInvalidJSON      = "INVALID_JSON"
    ErrStrategyInvalid  = "STRATEGY_INVALID"
    ErrInstanceNotRunning= "INSTANCE_NOT_RUNNING"
    ErrFileNotFound     = "FILE_NOT_FOUND"
    ErrConfigInvalid    = "CONFIG_INVALID"
)

// 统一替换所有 writeError 调用
// 旧: writeError(w, 404, "instance not found")
// 新:
errors.Respond(w, 404, errors.APIError{

统一状态查询接口 /state

一个请求返回 AI 需要的全部上下文：实例列表、运行状态、关键测点摘要
P1 · 高优先级 可行性：高 工作量：1 天
现状分析

AI 每次操作前需要了解模拟器状态，当前必须调 4 个端点拼凑：

AI 当前获取完整状态需要：
1. GET /api/v1/status           → 全局: running/stopped/max
2. GET /api/v1/instances        → 实例列表 + 各自 stats
3. GET /api/v1/instances/{id}/points  → 每实例测点快照
4. GET /api/v1/instances/{id}/metrics → 每实例指标

→ N 个实例 = 2 + 2N 次请求，AI context window 浪费严重
改造方案
// GET /api/v1/state — 一个请求获取全部上下文

      "port": 2404,

        "interrogations": 5,

幂等性 + 操作 ID

写操作返回 operation_id，重复调用同 ID 不重复执行——AI 重试零副作用
P1 · 高优先级 可行性：中 工作量：3-5 天
现状分析
  • 所有写操作（置数、创建实例、启停）无操作追踪，无去重
  • AI 调用超时后重试 → 可能重复置数、重复创建实例
  • batch-replay 已有 batch_id 进度追踪机制，是最接近的现有模式
  • 实例 ID 用 crypto/rand 生成（6 字节 hex），创建操作天然不幂等
改造方案

两层机制：客户端幂等键（Idempotency-Key header）+ 服务端操作日志。

// AI 发请求时带幂等键
PUT /api/v1/instances/a1b2c3/points/16385
Idempotency-Key: ai-task-001-set-voltage
Content-Type: application/json

{"value": 235.5}

// 服务端响应
{
  "success": true,
  "ioa": 16385,
  "operation_id": "op-7f3a2b",
  "idempotent_replay": false    // true = 命中缓存，未实际执行
}

// AI 超时重试，带相同 key → 直接返回上次结果，不重复写
PUT /api/v1/instances/a1b2c3/points/16385
Idempotency-Key: ai-task-001-set-voltage
{"value": 235.5}

→ {
  "success": true,
  "operation_id": "op-7f3a2b",
  "idempotent_replay": true     // 缓存命中
}
技术实现
// pkg/middleware/idempotency.go — 中间件拦截写操作
type IdempotencyStore interface {
    Get(key string) (cached []byte, ok bool)
    Set(key string, response []byte, ttl time.Duration)
}

// 内存实现（单进程够用），LRU + TTL 5分钟
type memoryStore struct {
    mu sync.RWMutex
    data map[string]*entry
}

func IdempotencyMiddleware(store IdempotencyStore) Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            // 仅拦截 POST/PUT/DELETE
            if r.Method == http.MethodGet { next.ServeHTTP(w, r); return }
            key := r.Header.Get("Idempotency-Key")
            if key == "" { next.ServeHTTP(w, r); return }
            // 检查缓存
            if cached, ok := store.Get(key); ok {
                w.Write(cached); return
            }
            // 捕获响应，缓存后写入
            recorder := newResponseRecorder(w)
            next.ServeHTTP(recorder, r)
            store.Set(key, recorder.Bytes(), 5*time.Minute)
        })
    }
}
范围控制：仅对写操作启用幂等中间件（POST/PUT/DELETE）。 GET 操作天然幂等不需要。实例创建（POST /instances）特殊处理：相同 Idempotency-Key 返回已创建的实例， 而非创建新实例。
文件影响
+ pkg/middleware/idempotency.go — 幂等中间件 + 内存 store (~200行)
~ cmd/gridsim/main.go — 中间件链插入 (+5行)
· Go 业务代码 — 零改动（中间件层透明拦截）

场景录制 / 回放

将 AI 操作序列固化为可重放的测试 fixture，回归测试一键复用
P2 · 中优先级 可行性：中 工作量：5-7 天
现状分析
  • ✓ 已有：CSV 数据回放（按时间序列回放测点值）
  • ✓ 已有：自动变化策略导出/导入（策略配置可序列化）
  • ✓ 已有：batch-replay 批量回放 + 进度追踪（batch_id + batchState）
  • 缺失：API 操作序列录制（记录 create/start/set-point/stop 的调用顺序）
  • 缺失：操作序列回放（按录制顺序重放，含时间间隔）
  • 缺失：场景断言（回放后验证测点值、状态是否符合预期）
改造方案

三层架构：录制器（中间件拦截记录）→ 存储（JSON 序列化）→ 回放器（按序重放 + 断言）。

主动事件流 (SSE)

模拟器内部状态变化主动推给 AI，替代低效轮询
P2 · 中优先级 可行性：中 工作量：4-6 天
现状分析

🔴 现状 — 前端轮询

MonitorPage:  setInterval 5000ms
MicrogridEditor: setInterval 2000ms + 3000ms
DetailPage:   setInterval 100-1000ms (可调)
TrendPanel:   setInterval (可调)

问题：
- 2秒轮询 = 30 req/min/客户端
- 空轮询浪费带宽 + context
- AI 无法及时感知变化
- 事件可能落在两次轮询之间被漏掉

🟢 SSE 推送

GET /api/v1/instances/{id}/events
Accept: text/event-stream

data: {"type":"point_changed","ioa":16385,"value":235.5}
data: {"type":"interrogation","client":"192.168.1.100"}
data: {"type":"control","ioa":5,"value":"ON"}
data: {"type":"client_connected","addr":"10.0.0.5:51234"}
data: {"type":"client_disconnected"}
data: {"type":"strategy_trigger","ioa":16385,"strategy":"increment"}

优势：
- 零空轮询
- 事件延迟 < 50ms
- AI 实时感知所有变化
- 单连接，低开销
为什么选 SSE 而非 WebSocket
  • 单向推送够用：模拟器→AI 是通知，AI→模拟器走 REST API，不需要双向通道
  • HTTP 原生：SSE 基于标准 HTTP，无需协议升级，穿透防火墙无问题
  • 自动重连：浏览器/HTTP 客户端原生支持 EventSource 重连
  • Go 标准库支持：net/http 直接 Flusher 即可，无需第三方库
  • AI Agent 友好：curl/HTTP 客户端直接消费 SSE 流，无需 WebSocket 库
技术实现
// pkg/events/bus.go — 进程内事件总线
type EventBus struct {
    mu       sync.RWMutex
    channels map[string][]chan Event  // key: instance_id
}

type Event struct {
    Type      string      `json:"type"`      // point_changed | interrogation | control | ...
    Timestamp time.Time   `json:"ts"`
    InstanceID string     `json:"instance_id"`
    Data      map[string]any `json:"data"`
}

// SSE handler
func HandleSSE(bus *EventBus) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        flusher, ok := w.(http.Flusher)
        if !ok { return }
        w.Header().Set("Content-Type", "text/event-stream")
        w.Header().Set("Cache-Control", "no-cache")
        w.Header().Set("Connection", "keep-alive")

        ch := bus.Subscribe(instanceID)
        defer bus.Unsubscribe(instanceID, ch)

        for {
            select {
            case event := <-ch:
                data, _ := json.Marshal(event)
                fmt.Fprintf(w, "data: %s\n\n", data)
                flusher.Flush()
            case <-r.Context().Done():
                return
            }
        }
    }
}
事件源头：IEC104 协议层的 publisher 接口（engine.go:15）已有 Publish(point *config.Point) 方法。只需在现有 Publish 调用链中加一个 EventBus 发布即可， 无需改协议层代码。遥控/总召唤事件在 pkg/iec104/ 的 handler 中添加事件发布点。
